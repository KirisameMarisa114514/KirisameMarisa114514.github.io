FastAPI + psutil + Chart.js：轻量系统监控仪表盘容器化

Created2026-07-19|Updated2026-07-19|开发云原生PythonDocker
|Word Count:951|Reading Time:3mins|Post Views:

这是一个基于 FastAPI、psutil 和 Chart.js 的轻量系统监控项目。后端负责采集 CPU、内存、磁盘、网络和进程数据，前端定时请求 REST API，再将数据绘制成动态图表。

项目结构

依赖非常精简：

FastAPI 同时提供 API 和静态页面，因此不需要再单独启动一个前端开发服务器。

API 设计

路径 数据
/api/cpu 逻辑核心数、物理核心数、CPU 使用率
/api/memory 总内存、已用内存、可用内存、使用率
/api/disks 分区、挂载点、文件系统与使用率
/api/network 累计流量与上传、下载速度
/api/processes 按 CPU 排序的进程列表

主应用挂载静态目录并返回首页：

这种结构适合小型仪表盘：部署单一、跨域问题少，也容易打包进一个容器。

前端可视化

前端使用 Chart.js 展示：

  • CPU 使用率折线图；
  • 内存占用饼图；
  • 网络上传与下载曲线；
  • 可搜索、可排序的进程表格；
  • 深色模式和响应式卡片布局。

前端轮询简单直观，但请求过密会增加采样开销。进一步优化时，可以让后端在后台统一采样，再通过 WebSocket 或 Server-Sent Events 推送给多个客户端。

容器监控的边界

默认情况下，容器中的 psutil 主要看到容器自己的进程和命名空间，它不天然等于“完整宿主机监控”。如果希望读取宿主机的全部进程、磁盘和网络信息，需要额外挂载 /proc、使用宿主机 PID 命名空间，或把采集器直接运行在宿主机。

这些方式会提高权限和攻击面，因此不能只为了数据更完整就盲目使用 privileged: true。更稳妥的选择是：

  1. 明确需要哪些指标；
  2. 只挂载必要的只读目录；
  3. 为监控页面增加认证；
  4. 不把进程列表直接暴露到公网；
  5. 为采集接口增加超时、缓存和访问频率限制。

后续改进

  • 加入磁盘 IO、温度和 GPU 指标；
  • 使用 WebSocket 推送；
  • 为历史数据接入 SQLite、Prometheus 或时序数据库；
  • 增加登录认证和反向代理；
  • 区分容器指标与宿主机指标；
  • 为异常阈值增加告警。

这个项目让我把 Python 数据采集、REST API、前端可视化和容器部署串成了一条完整链路，也更清楚地理解了容器隔离对监控数据的影响。
