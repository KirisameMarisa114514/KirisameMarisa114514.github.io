Docker Compose 部署 OpenList：持久化配置与本地目录挂载

Created2026-07-19|Updated2026-07-19|云原生Docker
|Word Count:821|Reading Time:2mins|Post Views:

OpenList 可以把不同来源的文件统一组织到一个 Web 界面中。这次实践使用 Docker Compose 部署 AIO 镜像，并把配置、运行数据和实际文件目录分开挂载，便于升级与备份。

当前部署结构

运行环境使用：

项目 当前配置
镜像 openlistteam/openlist:v4.1.9-aio
容器名 openlist
Web 端口 5244:5244
配置目录 ./config:/opt/openlist/config
数据目录 ./data:/opt/openlist/data
文件目录 ./downloads:/data

为什么要拆分挂载

config

config 保存程序级配置。把它持久化后，重建容器时不需要重新调整基础运行参数。

data

data 保存数据库、日志和 OpenList 自身的运行状态，是最需要备份的部分。数据库正在写入时不应直接随意复制；更稳妥的做法是先停止容器，或使用数据库支持的一致性备份方式。

downloads

downloads 是真正存放用户文件的宿主机目录，在容器内映射为 /data。OpenList 后台添加“本地存储”时，应填写容器看到的 /data，而不是宿主机路径。

这也是容器挂载最容易混淆的地方：

Web 管理界面运行在容器内部，因此它只能理解右侧路径。

添加本地存储时的检查项

  1. 驱动选择“本地存储”；
  2. 挂载路径使用站点中的虚拟路径；
  3. 根文件夹路径填写容器内的 /data；
  4. 检查目录的读取和写入权限；
  5. 不要把管理令牌、密码或数据库文件写进博客和公开仓库。

权限问题不建议通过整个项目递归 777 来长期解决。更合适的方式是确认容器运行用户，再为对应目录设置所有者和最小可用权限。

升级策略

升级前先记录当前镜像版本，并备份 config 与 data：

升级后至少验证：

  • 首页可以打开；
  • 管理员能够登录；
  • 存储列表正常；
  • /data 下的目录可以浏览；
  • 上传、下载和预览符合预期；
  • 日志中没有反复重启或数据库错误。

这次实践的收获

OpenList 的部署命令并不复杂，真正需要理解的是“宿主机、容器和应用配置”三层路径之间的关系。只要配置和数据已经持久化，容器本身就可以随时重建；而目录挂载、备份一致性和最小权限，才决定服务能否长期稳定运行。
