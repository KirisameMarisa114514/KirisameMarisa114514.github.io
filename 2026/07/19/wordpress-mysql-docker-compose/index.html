WordPress + MySQL：用 Docker Compose 搭建双容器博客

Created2026-07-19|Updated2026-07-19|开发云原生DockerWeb
|Word Count:900|Reading Time:3mins|Post Views:

WordPress 是学习 Docker Compose 多容器编排的经典场景：Web 应用依赖数据库，两者有独立生命周期，又必须通过内部网络互相通信。

这次部署使用两个服务：

服务 镜像 对外端口 持久化
WordPress wordpress:latest 8081:80 /var/www/html
MySQL mysql:8.0 不对宿主机公开 /var/lib/mysql

浏览器通过宿主机的 8081 端口访问 WordPress，而 WordPress 在 Compose 默认网络中通过服务名 db 连接 MySQL。

WORDPRESS_DB_HOST 使用的是 db:3306，不是 localhost。在 WordPress 容器内部，localhost 指向 WordPress 容器自己；Compose 服务名才是访问数据库容器的稳定地址。

启动流程

先执行 docker compose config 可以提前发现 YAML 缩进、变量缺失和字段拼写问题。

部署后访问：

第一次打开会进入 WordPress 初始化页面。站点管理员密码与数据库密码用途不同，应分别设置并妥善保存。

数据持久化

MySQL 数据保存在命名卷 db_data。即使删除并重建数据库容器，命名卷仍可保留数据。

查看卷：

WordPress 的程序、插件、主题和上传文件位于 /var/www/html。正式使用时也应为它配置明确的命名卷或绑定挂载，不能只依赖容器可写层。

服务依赖不等于服务就绪

depends_on 只控制启动顺序，并不保证 MySQL 已经完成初始化。WordPress 可能在数据库启动初期短暂连接失败。

备份策略

一个可恢复的 WordPress 备份至少包含两部分：

  1. MySQL 数据库；
  2. wp-content 中的主题、插件和上传文件。

数据库可以通过 mysqldump 导出，文件则使用归档或快照备份。只备份其中一部分，通常无法完整恢复站点。

恢复演练同样重要。备份文件存在不代表一定可用，应该定期在独立环境中验证导入流程。

上线前的安全检查

  • 不把 MySQL 3306 端口直接暴露到公网；
  • 使用 .env 或更安全的密钥管理方式；
  • 固定镜像版本，避免 latest 带来不可控升级；
  • 使用反向代理提供 HTTPS；
  • 定期更新 WordPress、主题与插件；
  • 限制后台登录并启用强密码；
  • 为数据库和 wp-content 建立自动备份；
  • 查看容器日志和磁盘占用。

这次实践的收获

WordPress + MySQL 虽然只有两个容器，却覆盖了 Compose 中最核心的概念：服务发现、内部网络、环境变量、命名卷、启动依赖和数据备份。理解这些基础后，再扩展反向代理、Redis、自动证书和持续部署就会自然很多。
